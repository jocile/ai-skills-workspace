# PLAN: 📑 Frontend de Documentação e Catálogo de Skills (Docusaurus)

## 1. Visão Geral Técnica

Este documento especifica a implementação de um portal de documentação estático baseado em Docusaurus (v3+). O sistema utiliza uma arquitetura de **Múltiplas Instâncias (Multi-instance)** para segmentar guias instrucionais (pasta `docs/`) de um catálogo visual de habilidades (pasta `skills/`).

Os arquivos Markdown contidos na pasta `skills/` possuem uma dupla responsabilidade:

1. Servir como fonte de metadados operacionais para agentes baseados no **Google Antigravity**.
2. Gerar de forma automatizada um catálogo visual em formato de cards no frontend React.

### 1.1. Pilha Tecnológica (Tech Stack)

* **Framework Base:** Docusaurus (React + MDX)
* **Ambiente de Execução:** Node.js v18+
* **Estilização:** CSS Modules (Nativo do Docusaurus)
* **Infraestrutura e CI/CD:** GitHub Actions + GitHub Pages (ou Vercel) com validação de Front-Matter (Fail-fast).

## 2. Contrato de Dados Híbrido (Markdown Front-Matter Spec)

Para que tanto o agente Google Antigravity quanto o Docusaurus processem os arquivos sem falhas, todo arquivo contido na pasta `skills/` DEVE conter obrigatoriamente os seguintes metadados no bloco YAML inicial.

### 2.1. O Schema da Skill

```yaml
---
name: string          # OBRIGATÓRIO (Antigravity): ID único da skill, sem espaços.
title: string         # OBRIGATÓRIO (Docusaurus): Nome legível e amigável para exibição na UI e menus.
description: string   # OBRIGATÓRIO (Ambos): Resumo das capacidades (Máx. 150 caracteres).
---

```

### 2.2. Exemplo de Arquivo em Conformidade (`skills/analisador-logs.md`)

```markdown
---
name: analisador-logs
title: Analisador de Logs
description: Monitora arquivos de log do sistema e identifica anomalias críticas usando padrões regex predefinidos.
---

# Documentação da Skill: Analisador de Logs

Aqui entra o texto descritivo livre explicando como editar os parâmetros desta skill e detalhes de implementação...

```

## 4. Engenharia de Componentes (The React Spec)

### 4.1. Configuração do Core (`docusaurus.config.js`)

O Docusaurus é instruído a rodar a pasta `docs/` no preset padrão e a pasta `skills/` como um plugin isolado, garantindo escopos de dados separados.

```javascript
module.exports = {
  // ... configurações padrão (title, url, baseUrl, etc.)

  presets: [
    [
      'classic',
      ({
        docs: {
          id: 'default', // Instância primária
          path: 'docs',
          routeBasePath: 'docs',
          sidebarPath: require.resolve('./sidebars.js'),
        },
      }),
    ],
  ],

  plugins: [
    [
      '@docusaurus/plugin-content-docs',
      {
        id: 'skills', // Identificador da instância secundária
        path: 'skills', 
        routeBasePath: 'skills', 
        sidebarPath: require.resolve('./sidebarsSkills.js'), 
      },
    ],
  ],
};

```

### 4.2. Navegação Automática (`sidebarsSkills.js`)

Para que a barra lateral da rota `/skills/` seja gerada automaticamente com base nos arquivos:

```javascript
module.exports = {
  skillsSidebar: [
    {
      type: 'autogenerated',
      dirName: '.', // Gera a sidebar com base na raiz da pasta /skills/
    },
  ],
};

```

### 4.3. Componente de Exibição Dinâmica (`src/components/SkillCards/index.js`)

Este componente consome a API global do Docusaurus extraindo dados EXCLUSIVAMENTE da instância `skills`.

```javascript
import React from 'react';
import Link from '@docusaurus/Link';
import { usePluginData } from '@docusaurus/useGlobalData';
import styles from './styles.module.css';

export default function SkillCards() {
  // Consome APENAS os dados da instância 'skills'
  const skillsData = usePluginData('docusaurus-plugin-content-docs', 'skills');
  
  // Extrai o array de documentos da versão atual
  const skills = skillsData?.versions?.[0]?.docs || [];

  if (skills.length === 0) {
    return <p>Nenhuma skill catalogada no momento.</p>;
  }

  return (
    <div className={styles.gridContainer}>
      {skills.map((skill) => (
        <div key={skill.id} className={styles.card}>
          <div className={styles.cardHeader}>
            <h3>{skill.title}</h3>
            <p>{skill.description}</p>
          </div>
          <Link className="button button--primary button--block" to={skill.path}>
            Ver Documentação de Edição
          </Link>
        </div>
      ))}
    </div>
  );
}

```

## 5. Protocolo de Integração na Interface (MDX Consumer)

Para disponibilizar o catálogo em qualquer página ou guia geral, o escritor deve utilizar a sintaxe MDX importando o componente:

```markdown
---
title: Hub de Inteligência
sidebar_position: 1
---
# Catálogo de Skills Disponíveis

Gerencie e edite o comportamento das nossas habilidades através das diretrizes abaixo.

import SkillCards from '@site/src/components/SkillCards';

<SkillCards />

```

## 6. Pipeline de Entrega Contínua (CI/CD) e Validação de Contrato

O deploy deve ser executado de forma agnóstica de infraestrutura local, disparado pelo arquivo `.github/workflows/deploy.yml`.

* **Trigger:** Push na branch `main`.
* **Sanity Check (Docusaurus):** Valida links quebrados nativamente durante o build (`npm run build`).
* **Validação do Contrato (Fail-fast):** A pipeline deve incluir uma etapa de script (ex: via Node.js ou `markdownlint`) que varra a pasta `skills/` ANTES do build, garantindo que as chaves `name`, `title` e `description` estão presentes no front-matter. Se falhar, o PR é bloqueado ou o deploy é interrompido.
